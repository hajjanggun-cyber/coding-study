건물의 정문(대문)

네이버: 메일 / 카페 / 블로그 / 뉴스 / 쇼핑

2. LNB (Local Navigation Bar)

뜻: 로컬 메뉴 바
위치: GNB 바로 아래
특징: 특정 서비스 안에 들어갔을 때만 보임

역할

해당 서비스 안에서의 큰 카테고리 이동

“뉴스 안의 정치/경제”, “쇼핑 안의 가전/식품” 같은 구조

비유

건물 안의 복도 안내판

예시

네이버 뉴스: 정치 / 경제 / 사회 / 문화

쿠팡: 로켓배송 / 로켓프레시 / 골드박스 / 카테고리 탭

3. SNB (Side Navigation Bar)

뜻: 사이드 메뉴 바
위치: 화면 좌측 또는 우측
특징: 세로 형태가 많음

역할

세부 메뉴, 필터, 관리 기능 제공

사용자 맞춤 메뉴가 많음

비유

엘리베이터 옆 층별 안내판

예시

블로그 관리: 글쓰기 / 통계 / 설정

쇼핑몰: 가격 필터 / 브랜드 필터 / 마이페이지 세부 메뉴

4. FNB (Footer Navigation Bar)

뜻: 푸터 메뉴 바
위치: 페이지 맨 아래
특징: 스크롤 끝까지 내려야 보임

역할

회사 정보, 약관, 고객센터, 사업자 정보

법적·공식 정보 영역

비유

명함 또는 건물 안내 책자

예시

랜딩 페이지(Landing Page
랜딩 페이지(Landing Page)**는 사용자가 광고나 검색 결과 등을 통해 웹사이트를 방문했을 때 가장 먼저 도달(Landing)하게 되는 페이지를 말합니다.

주로 마케팅 목적으로 사용되며, 방문자에게 특정 행동(회원가입, 제품 구매, 문의하기 등)을 유도하도록 설계된 페이지입니다.

일반적인 홈페이지가 기업의 전반적인 정보를 보여주는 '백과사전'이라면, 랜딩 페이지는 하나의 목표를 위해 만들어진 **'전단지'**나 **'카탈로그'**와 같습니다.

구체적인 특징과 구성 요소는 다음과 같습니다.

1. 주요 특징
단일 목표: 방문자가 다른 곳으로 새지 않고 오직 '버튼' 하나를 누르게 만드는 데 집중합니다.

간결한 구조: 복잡한 메뉴를 없애고, 위에서 아래로 흐르는 스토리텔링 방식을 주로 사용합니다.

강력한 메시지: 사용자에게 이 서비스가 왜 필요한지, 어떤 혜택이 있는지 한눈에 보여줍니다.

2. 핵심 구성 요소
히어로 섹션 (Hero Section): 가장 상단에 위치하며, 서비스의 첫인상을 결정하는 큰 이미지와 핵심 문구입니다.

기능 및 혜택 (Features & Benefits): 제품의 강점을 설명하고 사용자의 문제를 어떻게 해결해 주는지 보여줍니다.

사회적 증거 (Social Proof): 실제 사용자의 후기나 파트너사의 로고 등을 통해 신뢰를 줍니다.

CTA 버튼 (Call To Action): '지금 시작하기', '무료로 체험하기'와 같이 최종 행동을 유도하는 눈에 띄는 버튼입니다.

💡 사용자님의 프로젝트와 연관 짓기
사용자님이 앞서 정리하신 유튜브 영상 내용을 보면, 가장 먼저 만든 것이 바로 이 **'랜딩 페이지'**였습니다.

RAG는 Retrieval-Augmented Generation
RAG는 Retrieval-Augmented Generation의 약자입니다.
RAG는 보통 “검색 증강 생성”이라고 많이 부릅니다.

검색을 강화해서 답을 만들어내는 방식이라는 뜻입니다.

쉽게 말하면 AI가 먼저 자료를 찾고 그 다음에 답을 쓰는 구조

RAG는 Retrieval-Augmented Generation
1. Retrieval (리트리벌)
영어 뜻: 검색, 회수, 되찾아옴.

의미: 방대한 데이터 저장소(사용자님의 구글 시트 등)에서 질문과 관련된 특정 정보를 찾아내는 과정을 말합니다.

한국식 발음: 리트리벌 또는 리트리버 (강아지 '골든 리트리버'의 그 리트리버와 같은 어원입니다.)

2. Augmented (오그멘티드)
영어 뜻: 증강된, 확장된, 강화된.

의미: 원래의 질문에 검색된 정보를 덧붙여서 내용을 더 풍부하게 만든다는 뜻입니다.

한국식 발음: 오그멘티드

3. Generation (제너레이션)
영어 뜻: 생성, 발생.

의미: 강화된 정보를 바탕으로 AI 모델이 최종적인 답변을 만들어내는 단계입니다.

한국식 발음: 제너레이션


RAG는 Retrieval-Augmented Generation의 약자고, 한국어로는 보통 **“검색 결합 생성”**이라고 불러요.
한 문장으로는:

LLM이 답을 ‘그냥 기억에서’ 만들게 두지 않고, 먼저 관련 문서를 찾아(검색) 그 근거를 붙여서 답을 생성하게 만드는 방식이에요.

왜 RAG가 필요한가

LLM은 똑똑하지만 한계가 있어요.

최신 정보/사내 문서/내 파일 같은 건 모델이 원래 모르거나 학습에 없을 수 있음

자신감 있게 틀리는 환각(헛소리) 가능

출처/근거 없이 말하면 검증이 어려움

RAG는 이걸 이렇게 해결해요:

내 문서/DB/위키/스프레드시트에서 관련 내용을 뽑아오고

그 내용을 “컨텍스트”로 LLM에 넣어서

근거 기반 답변을 만들게 함

Context Window / Context Engineering
1. 컨텍스트 윈도우 (Context Window)
컨텍스트 윈도우는 인공지능이 한 번에 처리하고 기억할 수 있는 데이터의 최대 양을 의미합니다.

정의: 대화 도중 모델이 '주의'를 기울여 읽을 수 있는 텍스트의 범위입니다.

단위: 보통 단어보다 작은 단위인 **토큰(Token)**으로 측정됩니다.

작동 방식: 대화가 길어져 이 윈도우 범위를 넘어가면, 모델은 대화의 앞부분(과거 내용)을 잊어버리기 시작합니다.

영향: 컨텍스트 윈도우가 클수록 수천 페이지의 책이나 복잡한 코드 전체를 한 번에 입력하고 분석할 수 있습니다.

2. 컨텍스트 엔지니어링 (Context Engineering)
제한된 컨텍스트 윈도우 안에서 모델이 최고의 성능을 내도록 데이터를 전략적으로 구성하는 기술입니다.

정의: 모델에게 전달하는 정보(프롬프트, 외부 데이터 등)를 최적화하여 답변의 정확도를 높이는 과정입니다.

주요 기법:

프롬프트 엔지니어링: 질문의 순서나 형식을 조정하여 모델이 맥락을 더 잘 파악하게 합니다.

RAG (검색 증강 생성): 방대한 데이터 중 질문과 가장 관련 있는 부분만 요약해서 컨텍스트 윈도우에 집어넣는 방식입니다.

압축 및 요약: 이전 대화 내용을 핵심만 추려 전달함으로써 토큰 낭비를 줄입니다.


나쁜 방식

Prompt Engineering / Context Engineering
사용자님께서 앞서 다루신 '코딩 용어 정리 시스템'에 비유하여 그 차이를 명확히 정리해 드립니다.

1. 프롬프트 엔지니어링 (Prompt Engineering)
사용자가 AI에게 직접 전달하는 **'지시문의 품질'**에 집중하는 기술입니다.

핵심: 모델의 내장된 추론 능력을 극대화하기 위해 질문의 형식을 다듬는 것입니다.

주요 기법:

역할 부여 (Persona): "너는 전문적인 프로그래머야"와 같은 페르소나 설정.

단계별 사고 (CoT): "차근차근 생각해보자"와 같은 문구를 추가하여 논리적 답변 유도.

출력 형식 지정: "JSON 형식으로 답변해줘"와 같은 구조적 요청.

비유: 학생(AI)에게 시험 문제를 낼 때, 의도가 명확하도록 문제를 아주 정교하고 상세하게 출제하는 것과 같습니다.

2. 컨텍스트 엔지니어링 (Context Engineering)
AI가 답변을 생성할 때 참고할 수 있는 **'기억과 배경 정보'**를 설계하는 기술입니다.

핵심: 제한된 컨텍스트 윈도우 내에서 모델이 가장 정확한 정보를 참조할 수 있도록 데이터를 주입하고 관리하는 시스템적 접근입니다.

주요 기법:

RAG (검색 증강 생성): 수만 개의 시트 데이터 중 질문과 가장 관련 있는 데이터만 골라 AI에게 전달하는 기술.

토큰 관리: 대화 이력을 요약하거나 불필요한 데이터를 삭제하여 효율적으로 컨텍스트 윈도우를 활용하는 기술.

데이터 구조화: AI가 정보를 쉽게 읽도록 데이터를 계층화하거나 관련성 있게 배치하는 작업.

비유: 학생(AI)이 시험을 볼 때 옆에 놓아주는 **'최고의 요약 노트'**를 구성하는 것과 같습니다.

Node(연결되는 점(지점)) / Node.js / npm
- Node : 연결되는 점(지점)
1. 네트워크(Node) :  컴퓨터나 서버 같은 연결된 장치 하나를 말합니다.
인터넷에 연결된 내 PC = 하나의 노드 / 서버 한 대 = 하나의 노드

2. 자료구조(Node) : 데이터 구조(트리, 리스트 등)에서 데이터가 들어 있는 하나의 칸
3. Node.js 에서의 Node 여기서는 고유명사처럼 쓰임
자바스크립트를 컴퓨터에서 실행하게 해주는 프로그램 이름입니다.

웹브라우저 없이도 JS 실행 가능 / 서버 개발 가능 /  👉 자바스크립트 실행 엔진(비유)


- Node.js = 자바스크립트를 컴퓨터에서 실행하게 해주는 프로그램

자바스크립트(JS)는 원래 웹 브라우저 전용 언어였어요.
크롬, 엣지, 사파리 이런 브라우저 안에서만 실행됐습니다.
그런데 Node.js가 나오면서 “브라우저 없이도 자바스크립트를 컴퓨터에서 직접 실행 가능”

왜 claude 설치할 때 Node.js가 필요하냐?
claude CLI 같은 프로그램들은 자바스크립트로 만들어진 도구입니다.

그래서, Node.js = 자바스크립트 실행기
Node.js를 설치하면 그 설치 파일 안에 이미 npm이 있음

- npm(Node Package Manager) = Node.js용 프로그램 설치·삭제·업데이트 관리 도구라는 뜻
자바스크립트 앱 설치기 즉 프로그램을 찾아서 설치하는 앱 마켓 역할을 함.
내 컴퓨터의 npm이 → 인터넷 npm 창고에 접속해서 → 프로그램을 내려받아 → 내 PC에 설치합니다.

Node.js와 npm 두 개가 있어야 claude를 설치하고 실행할 수 있습니다.

Node.js를 다운로드 : 사용자가 직접 받는 것은 npm이 아니라 Node.js입니다.

GNB / LNB / SNB / FNB
웹사이트 구조를 이해할 때 GNB / LNB / SNB / FNB는 거의 기본 4대 메뉴 구조라고 보시면 됩니다.
조금 더 쉽게 다시 한 번 “위치 + 역할 + 비유”로 정리해 드릴게요.

1. GNB (Global Navigation Bar)

뜻: 전역 메뉴 바
위치: 페이지 최상단
특징: 어떤 페이지로 이동해도 항상 보임

역할

사이트 전체 서비스로 이동하는 가장 큰 길

브랜드 로고, 검색창, 로그인, 장바구니 같은 핵심 기능 포함

비유

건물의 정문(대문)

예시

네이버: 메일 / 카페 / 블로그 / 뉴스 / 쇼핑

쿠팡: 로고 / 검색창 / 로그인 / 장바구니 / 마이쿠팡

2. LNB (Local Navigation Bar)

뜻: 로컬 메뉴 바
위치: GNB 바로 아래
특징: 특정 서비스 안에 들어갔을 때만 보임

역할

해당 서비스 안에서의 큰 카테고리 이동

“뉴스 안의 정치/경제”, “쇼핑 안의 가전/식품” 같은 구조

비유

건물 안의 복도 안내판

예시

네이버 뉴스: 정치 / 경제 / 사회 / 문화

쿠팡: 로켓배송 / 로켓프레시 / 골드박스 / 카테고리 탭

3. SNB (Side Navigation Bar)

뜻: 사이드 메뉴 바
위치: 화면 좌측 또는 우측
특징: 세로 형태가 많음

역할

세부 메뉴, 필터, 관리 기능 제공

사용자 맞춤 메뉴가 많음

비유

엘리베이터 옆 층별 안내판

예시

블로그 관리: 글쓰기 / 통계 / 설정

쇼핑몰: 가격 필터 / 브랜드 필터 / 마이페이지 세부 메뉴

4. FNB (Footer Navigation Bar)

뜻: 푸터 메뉴 바
위치: 페이지 맨 아래
특징: 스크롤 끝까지 내려야 보임

역할

회사 정보, 약관, 고객센터, 사업자 정보

법적·공식 정보 영역

비유

명함 또는 건물 안내 책자

예시

이용약관 / 개인정보처리방침

사업자 등록번호 / 고객센터 번호 / SNS 아이콘

컨텍스트(Context
AI와 프로그래밍 분야에서 **컨텍스트(Context)**는 우리말로 '맥락' 또는 **'상황 정보'**라고 번역됩니다. 단순히 단어의 뜻을 넘어, **"시스템이나 모델이 현재 작업을 올바르게 처리하기 위해 알고 있어야 하는 주변 정보의 집합"**을 의미합니다.

분야별로 조금 더 구체적으로 설명해 드릴게요.

1. LLM(대화형 AI)에서의 컨텍스트
AI와 대화할 때 가장 중요한 개념입니다. AI는 사용자가 이전에 한 말이나 설정해둔 정보를 기억해야 흐름에 맞는 답변을 할 수 있는데, 이를 **'컨텍스트를 유지한다'**고 합니다.

컨텍스트 윈도우 (Context Window): AI가 한 번에 기억하고 처리할 수 있는 정보의 양입니다. 이 공간이 클수록 긴 문서나 많은 대화 기록을 한꺼번에 이해할 수 있습니다.

프롬프트 컨텍스트: 질문을 던질 때 "너는 전문 블로거야"라고 역할을 부여하거나 관련 데이터를 미리 주는 것도 컨텍스트를 제공하는 행위입니다.

2. 프로그래밍에서의 컨텍스트
코드 실행에 필요한 **'환경 정보'**를 뜻합니다.

Execution Context (실행 컨텍스트): 코드가 실행될 때 필요한 변수, 함수 선언, 스코프 등을 담고 있는 정보 덩어리입니다.

Android/Spring Context: 프레임워크에서 앱의 현재 상태나 시스템 자원에 접근하기 위한 통로 역할을 합니다.

3. MCP(Model Context Protocol)에서의 의미
앞서 질문하신 MCP에서의 컨텍스트는 **"AI 모델이 외부 데이터(내 로컬 파일, DB, 웹 문서 등)를 마치 자신의 기억 속에 있는 것처럼 실시간으로 가져와서 참고하는 환경"**을 뜻합니다.

이해를 돕기 위한 비유
우리가 친구와 카페에서 대화를 한다고 가정해 봅시다.

대화 내용: "그거 어땠어?"

컨텍스트: 우리가 어제 같이 본 영화, 지금 카페의 분위기, 친구의 기분 상태 등.

만약 컨텍스트가 없다면 AI는 "그게 뭔데?"라고 되물을 수밖에 없지만, 컨텍스트가 있다면 "어제 본 영화는 정말 감동적이었지!"라고 대답할 수 있게 됩니다.

저에게 명령하실 때는 어렵게 전문 용어를 쓰지 않으셔도 됩니다. **"어떤 상황에서, 어떻게 보였으면 좋겠는지"**만 구체적으로 말씀해 주시면 제가 알아서 코드로 바꿉니다.

가장 효과적인 '저를 부리는 주문' 3가지를 알려드릴게요.

1단계: 가장 쉬운 주문 (초급)
"이거 핸드폰으로 보면 글씨가 너무 작아. 모바일에서도 잘 보이게 해줘."

해석: "아, 반응형(Responsive) 작업을 해달라는 거구나. 글자 크기랑 여백을 자동으로 조절해야지."
팁: 그냥 '잘 보이게 해줘'라고 하면 제가 알아서 적당히 예쁘게 만듭니다.
2단계: 구체적인 주문 (중급)
"PC에서는 가로로 쭉 나열되는데, 폰에서는 세로로 한 줄씩 나오게 해줘."

해석: "레이아웃(배치)을 바꾸라는 거구나. PC는 가로(Flex Row), 모바일은 세로(Flex Column)로 설정해야지."
팁: '가로/세로' 같은 방향을 콕 집어 말해주면 제가 훨씬 정확하게 작업합니다.
3단계: 전문가처럼 보이고 싶을 때 (고급)

HTML 단추 용어2
1️⃣ 글자를 입력하는 상자들 (Input / Textarea / Placeholder)
설명: 빈칸 채우기랑 같아요.

인풋 (Input): 아이디나 제목처럼 한 줄만 짧게 쓰는 칸.
텍스트에어리어 (Textarea): 일기장처럼 엔터 쳐가며 여러 줄을 쓰는 큰 칸.
플레이스홀더 (Placeholder): 입력하기 전에 회색 글씨로 "여기에 입력하세요" 하고 써 있는 안내 문구.
🗣️ 이렇게 말해보세요:

"제목 적는 인풋(Input) 칸의 길이를 좀 더 늘려주세요." "내용 적는 텍스트에어리어(Textarea) 칸이 글을 많이 쓰면 자동으로 늘어나게 해주세요." "아무것도 안 썼을 때 보이는 플레이스홀더(Placeholder) 문구를 '이름을 입력하세요'로 바꿔주세요."

2️⃣ 마우스를 올렸을 때 효과 (Hover)
설명: 버튼 위에 마우스 화살표를 살짝 올려두는 동작을 말해요. 보통 색깔이 변하거나 밑줄이 생기죠.

🗣️ 이렇게 말해보세요:

"저장 버튼에 마우스를 올렸을 때(호버(Hover) 했을 때) 색깔이 파란색에서 빨간색으로 바뀌게 해주세요." "목록에 호버하면 배경색이 살짝 어두워지게 해주세요."

3️⃣ 껐다 켰다 하는 버튼 (Toggle)
설명: 전등 스위치랑 똑같아요. 한 번 누르면 켜지고(ON), 다시 누르면 꺼지는(OFF) 버튼을 말해요. 우리 프로그램의 [전체 선택] 버튼 같은 거죠.

🗣️ 이렇게 말해보세요:

"전체 선택 버튼을 누를 때마다 체크가 됐다가 풀렸다가 하게, 토글(Toggle) 방식으로 만들어주세요." "강의 내용을 접었다 폈다 하는 토글 버튼을 달아주세요."

4️⃣ 툭 튀어나오는 안내창 (Modal / Popup)
설명: 우리 프로그램 처음에 비밀번호 물어보는 창이나, 카테고리 이동할 때 뜨는 창 기억나시죠? 배경은 어두워지고 가운데 툭 튀어나와서 사용자의 시선을 뺏는 창을 말해요.

🗣️ 이렇게 말해보세요:

"비밀번호를 틀리면 경고창(알림창) 말고, 예쁜 모달(Modal)창을 띄워주세요." "이동 버튼을 누르면 카테고리를 고를 수 있는 모달이 뜨게 해주세요."

5️⃣ 핸드폰에서도 잘 보이는 것 (반응형 / Responsive)
설명: 컴퓨터 모니터는 넓고 핸드폰 화면은 좁죠? 물이 컵 모양에 따라 변하듯, 화면 크기에 맞춰서 디자인이 알아서 척척 바뀌는 걸 말해요.

🗣️ 이렇게 말해보세요:

"핸드폰으로 볼 때는 글씨가 너무 작아요. **모바일 반응형(Responsive)**으로 글자 크기를 조절해주세요." "반응형이라서 창을 줄이면 버튼들이 아래로 내려가네요? 아주 좋아요."

6️⃣ 새로고침 없이 바로 바뀌는 것 (동적 / AJAX)
설명: 옛날 사이트는 버튼 누르면 화면이 깜빡! 하고 다시 그려졌는데, 우리 프로그램은 저장 누르면 깜빡임 없이 리스트에 쑉! 하고 추가되죠? 이걸 어려운 말로 '동적이다' 또는 '비동기 처리'라고 해요.

🗣️ 이렇게 말해보세요:

"저장 버튼 눌렀을 때 페이지 새로고침 되지 않고, 동적으로 리스트만 갱신되게 해주세요." "삭제하고 나서 기다리지 않고 바로 화면에서 사라지게(즉시 반영) 해주세요."

✨ 총정리 실습 (한번 따라 읽어보세요!)
"개발자님, 상단 **탭(Tab)**을 눌렀을 때 반응 속도가 좀 느린 것 같아요. 그리고 아래 **리스트(List)**에 있는 **카드(Card)**에 마우스를 **호버(Hover)**하면 배경색을 좀 바꿔주세요. 아참, 수정은 페이지 이동 없이 **인라인(Inline)**으로 고칠 수 있게 해주시고, 핸드폰에서도 잘 보이게 **반응형(Responsive)**으로 신경 써주세요!"

이렇게 말씀하시면 개발자가 감동의 눈물을 흘릴지도 모릅니다. 😂

1. 상단 메뉴 영역 → "탭 (Tab)" 또는 "내비게이션 (GNB)"
스크린샷 맨 위, 파란색으로 선택된 [손님내용] 처럼 나열된 메뉴들을 말합니다.

탭 (Tab): 서류철의 인덱스 탭처럼 톡 튀어나와서 클릭하면 화면이 바뀌는 버튼들입니다.
내비게이션 바 (Navigation Bar/GNB): 사이트의 길을 안내해준다는 뜻으로, 전체 메뉴 줄을 통틀어 부를 때 씁니다.
예시: "상단 탭의 순서를 바꾸고 싶어요."
2. 내용 입력하는 곳 → "인풋(Input)"과 "텍스트에어리어(Textarea)"
중간에 글자를 적는 하얀색 박스들입니다.

인풋 (Input): '손님 제목'처럼 한 줄만 짧게 입력하는 칸입니다.
텍스트에어리어 (Textarea): '손님 내용'처럼 엔터 쳐가며 여러 줄을 길게 쓸 수 있는 큰 박스입니다.
플레이스홀더 (Placeholder): 입력하기 전에 회색으로 희미하게 보이는 안내 문구(예: 손님 제목 (예: 홍길동))를 말합니다. 타자를 치면 사라집니다.
3. 조작 버튼들 → "툴바 (Toolbar)" 또는 "컨트롤 (Control)"
하단에 [전체 선택], [선택 삭제] 등이 모여 있는 줄입니다.

체크박스 (Checkbox): ☑ 모양의 네모칸. 여러 개를 동시에 선택할 때 씁니다.
정렬 버튼 (Reorder Buttons): ⏫ 🔼 🔽 ⏬ 아이콘이 있는 버튼들입니다. 순서를 위아래로 바꿀 때 씁니다.
토글 버튼 (Toggle): 한 번 누르면 켜지고, 다시 누르면 꺼지는 스위치 같은 버튼입니다. (예: [전체 선택] 버튼)
4. 데이터 목록 → "카드 (Card)" 또는 "리스트 (List)"
맨 아래 Bar 두산동 방3개짜리... 처럼 데이터가 한 줄씩 표시되는 부분입니다.

카드 (Card): 하나의 정보 덩어리를 카드처럼 네모나게 묶어서 보여주는 디자인을 말합니다.
리스트 아이템 (List Item): 목록의 한 줄, 한 줄을 말합니다.
5. 핵심 기능 용어 → "인라인 수정 (Inline Edit)"
사용자님이 가장 궁금해하셨던 부분입니다.

인라인 수정 (Inline Edit):
보통 수정 버튼을 누르면 '수정 페이지'로 화면이 넘어가죠?
하지만 지금 프로그램처럼 화면 이동 없이 그 자리(Line) 안(In)에서 바로 글자가 입력창으로 변해서 고칠 수 있는 방식을 말합니다. 아주 세련되고 편리한 기능입니다.
예시: "목록에서 바로 고칠 수 있게 인라인 수정 기능 넣어주세요."

"상단 탭에서 손님내용을 클릭하고, 인풋창에 제목을 넣은 뒤 등록하면 아래 리스트에 추가됩니다. 목록에 있는 카드의 수정 버튼을 누르면 인라인으로 바로 고칠 수 있고, 체크박스를 켜서 정렬 버튼으로 위아래 순서를 바꿀 수 있습니다."
